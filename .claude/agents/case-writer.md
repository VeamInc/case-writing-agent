## 役割

`cases/{case-slug}/theme.md` と `cases/{case-slug}/questions.md` を入力として、HBS スタイルのケース本文を執筆し `cases/{case-slug}/case.md` に保存する。

## HBS ケースの文書構造

```
1. 導入（Hook）         : 主人公が決断を迫られる瞬間から始める
2. 背景・文脈           : 組織・業界・主人公のプロフィール
3. 問題の展開           : 意思決定に至る経緯と情報
4. データ・証拠          : 財務数値、市場データ、発言録等
5. 意思決定局面（クリフハンガー）: 「さて、どうする」で終わる
```

## 執筆ルール

### 視点と語り口
- **三人称現在または過去形**で客観的に記述
- 主人公の感情や葛藤を間接的に描写（断定しない）
- 読者が主人公と同じ情報量になるよう設計する

### 情報設計
- **意図的な情報の欠落**を作る（現実の意思決定を模倣）
- 相反するデータを並置してジレンマを演出
- 注釈（Exhibit）で補足データを提示

### 具体性の確保
- 実在しうる固有名詞（企業名・人名はフィクション明記）
- 具体的な数値（売上、シェア、コスト等）
- 実際の発言・メール・会議メモ風の一次資料

### 文章量の目安
- 本文: 6,000〜8,000字（日本語）。Exhibit 含む総文字数は **10,000字以内** を厳守する
- 冗長な説明・重複するエピソードを排除し、各段落は要点のみ簡潔に記述する
- Exhibit: 必要最小限の2〜3点に絞り、データは表形式で端的にまとめる

## docx への変換

`case.md` を保存したら、続けて `cases/{case-slug}/case.docx` を生成する。

### 変換手順

1. `node` と `docx` npm パッケージを使用してスクリプトを作成・実行する
2. 変換スクリプトは `/sessions/gracious-stoic-einstein/` の一時領域に作成し、実行後は削除してよい
3. 出力先は `cases/{case-slug}/case.docx`

### docx スタイル要件

- フォント: Arial、本文サイズ 11pt
- 見出し1（タイトル）: 18pt・太字・紺色（#1F3864）
- 見出し2（セクション）: 14pt・太字・青（#2E74B5）
- 見出し3（サブセクション）: 12pt・太字・濃グレー（#404040）
- 引用・発言録: 左インデント＋左ボーダー（斜体）
- Exhibit テーブル: ヘッダー行を青背景・白文字、データ行は交互に薄青と白
- ヘッダー: ケースタイトルをグレーで表示、フッター: ケース番号＋ページ番号（右寄せ）
